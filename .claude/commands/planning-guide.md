# Planning Guide

웹 서비스 기획 · PRD · 설계 작업 시 따르는 가이드.

---

## 기획 문서 구조 (PRD)

PRD는 항상 아래 순서로 작성한다:

1. **개요** — 한 줄 목적 + 배경
2. **목표 / 비목표** — 이번 범위에서 하는 것 / 안 하는 것 명확히 구분
3. **사용자 및 시나리오** — 누가, 언제, 왜 쓰는지
4. **기능 요구사항** — 우선순위(P0/P1/P2)와 함께 나열
5. **비기능 요구사항** — 성능, 보안, 접근성 기준
6. **UI 플로우** — 사용자 흐름을 텍스트 또는 ASCII 다이어그램으로 표현
7. **열린 질문** — 아직 결정 안 된 사항

## 아키텍처 설계 원칙

- 기술 선택 전에 **요구사항 → 제약 → 옵션 비교** 순서를 지킨다
- 다이어그램은 레이어 단위로: Client → API → Service → DB
- 외부 의존성(외부 API, 서드파티)은 명시적으로 경계를 표시한다
- 확장성보다 **현재 요구사항에 맞는 단순한 설계** 우선

## 유저 스토리 작성 규칙

```
As a [사용자 유형],
I want to [목표],
So that [이유/가치].

Acceptance Criteria:
- [ ] 조건 1
- [ ] 조건 2
```

## 우선순위 기준

| 등급 | 기준 |
|------|------|
| P0 | 없으면 서비스 불가 — 런칭 블로커 |
| P1 | 핵심 사용자 경험에 직접 영향 |
| P2 | 있으면 좋지만 없어도 출시 가능 |
| P3 | 추후 개선 백로그 |

## 금지 사항

- 기술 스택 결정을 기획 단계에서 고정하지 않는다
- "나중에 추가하면 됨" 표현 없이 P2/P3으로 명시한다
- 모호한 표현("빠르게", "쉽게") 사용 금지 — 수치로 정의
