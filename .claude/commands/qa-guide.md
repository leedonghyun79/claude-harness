# QA Guide

버그 수정 · 테스팅 · QA 작업 시 따르는 가이드.

---

## 버그 처리 프로세스

1. **재현** — 버그를 100% 재현하는 최소 케이스 확인
2. **원인 파악** — 증상이 아닌 근본 원인(root cause) 찾기
3. **범위 확인** — 동일 패턴의 다른 버그 존재 여부 확인
4. **수정** — 최소 변경으로 수정 (주변 코드 리팩토링 금지)
5. **테스트** — 수정 확인 + 회귀 테스트

## 테스트 작성 원칙

- 테스트는 **구현이 아닌 동작**을 검증한다
- 테스트명은 "무엇을 할 때 무엇이 되어야 한다" 형식:
  ```
  "로그인 실패 시 에러 메시지를 표시한다"
  "장바구니에 동일 상품 추가 시 수량이 증가한다"
  ```
- 각 테스트는 독립적으로 실행 가능해야 한다 (순서 의존 금지)
- 테스트 데이터는 테스트 내에서 생성하고 정리한다

## 테스트 유형별 가이드

### 단위 테스트 (Unit)
- 순수 함수 / 유틸리티 / 비즈니스 로직 검증
- 외부 의존성은 mock 처리
- 빠르게 실행되어야 함 (100ms 이하)

### 통합 테스트 (Integration)
- API 엔드포인트 요청/응답 검증
- 실제 DB 사용 (mock DB 지양)
- 인증 플로우, 데이터 흐름 검증

### E2E 테스트
- 핵심 사용자 시나리오만 커버 (전체 기능 X)
- 결제, 회원가입, 핵심 기능 플로우 위주

## 회귀 방지 규칙

- 버그 수정 시 해당 버그를 재현하는 테스트를 반드시 함께 추가
- 수정 전 실패 → 수정 후 통과 확인
- CI에서 테스트 통과 없이 머지 금지

## QA 체크리스트

- [ ] 정상 케이스(happy path) 동작 확인
- [ ] 엣지 케이스 (빈 값, 최대값, 특수문자) 확인
- [ ] 에러 상황 (네트워크 실패, 권한 없음) 처리 확인
- [ ] 모바일 / 다른 브라우저에서 동작 확인
- [ ] 로딩/비어있는 상태 UI 확인

## 금지 사항

- 원인 파악 없이 증상만 숨기는 수정 (band-aid fix)
- `// TODO: fix later` 남기고 커밋
- 실패하는 테스트를 skip/ignore 처리
