## 장애 대응 프로세스

### 1단계 — 감지
- 모니터링 알림 또는 사용자 제보 확인
- 브라우저 콘솔 에러 확인
- 서버 로그 확인

### 2단계 — 격리
- 영향 범위 파악 (전체 서비스? 특정 기능?)
- 최근 배포 이력 확인 (`git log --oneline -10`)
- 최근 배포가 원인이면 즉시 롤백 (롤백 방법은 [operations/deploy.md](operations/deploy.md) 참고)

### 3단계 — 복구
- 롤백으로 해결 가능 → 즉시 롤백
- 코드 수정 필요 → hotfix 브랜치 → PR → 배포

### 4단계 — 기록
- `docs/exec-plans/active/incident-{날짜}.md` 작성
- 원인, 영향 범위, 복구 시간, 재발 방지 방안 포함

## 신뢰성 체크리스트

### 에러 처리
- [ ] 모든 API route에 try-catch 및 에러 응답 처리가 있는가?
- [ ] 클라이언트에 스택 트레이스가 노출되지 않는가?
- [ ] 네트워크 실패 시 사용자에게 적절한 피드백을 주는가?
- [ ] 빈 상태(empty state), 로딩 상태 UI가 구현되어 있는가?
- [ ] 중복 제출(double submit) 방지 처리가 있는가?

### 모니터링
- [ ] 에러 트래킹(Sentry 등)이 연동되어 있는가?
- [ ] 배포 후 핵심 기능 동작을 수동 확인하는가?
- [ ] 비정상 트래픽/에러율 급증 알림이 있는가?

### 복구 가능성
- [ ] 이전 버전으로 즉시 롤백 가능한가?
- [ ] DB 마이그레이션은 되돌릴 수 있는가?
- [ ] 배포 전 스테이징 환경에서 검증하는가?

### 데이터 무결성
- [ ] 중요 데이터 변경 전 트랜잭션을 사용하는가?
- [ ] 낙관적 업데이트(optimistic update) 실패 시 롤백되는가?
